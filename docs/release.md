# Release flow

This app releases automatically on every push to `master`. There is no manual tag-push step in the normal path.

## PR-title convention

The release workflow buckets each commit subject under a conventional-commit prefix to build `CHANGELOG.md`. To land in a named bucket, use one of these prefixes (optionally scoped, optionally with `!` for breaking):

- `feat:` — Features
- `fix:` — Fixes
- `refactor:` — Refactors
- `perf:` — Performance
- `docs:` — Docs
- `chore:`, `test:`, `ci:`, `build:` — Other Changes
- anything else (including no prefix) — Other Changes

Regex enforced by the advisory PR-title check:

```
^(feat|fix|refactor|perf|docs|chore|test|ci|build)(\([^)]+\))?!?: 
```

The check is **advisory**: it posts a comment on the PR but does not block CI. The build still runs and the merge still happens. The comment dedupes by a marker containing a hash of the title — re-renaming and re-pushing without changing the title is a no-op.

## Auto-release flow

1. A PR is merged to `master` (with a conventional-commit title).
2. The `.github/workflows/release.yml` workflow fires on the push.
3. `bash bin/release-tag` computes the next tag:
   - HEAD is already tagged → return that tag (no-op).
   - Otherwise resolve the baseline as `max(latest_tag_version, pubspec_version)`, derive the bump kind from the merge commit's subject (`feat:` → minor, everything else → patch), and tag HEAD with `v<next>`.
4. The new tag is pushed to `origin`. That push triggers the existing `.github/workflows/android_release.yml`, which builds and publishes the APK to the GitHub release.
5. Back in `release.yml`, `pubspec.yaml` is bumped to `<new_version>+<build+1>` and committed with the bot identity.
6. `CHANGELOG.md` is regenerated by `bash bin/generate-changelog` and committed with `[skip ci]` so it does not re-trigger the workflow.
7. Both commits are pushed to `master`.

The APK release is **not** produced by `release.yml` itself. It comes from `android_release.yml` reacting to the tag push.

## Pubspec / tag alignment contract

Pubspec's `X.Y.Z` part is the seed for the next tag's baseline. The resolver picks the higher of the latest tag's version and the current pubspec version. After every release, the workflow rewrites pubspec to the new version with a bumped build number, so:

- `pubspec.yaml`'s `version: X.Y.Z+N` and the most recent `vX.Y.Z` tag are in lockstep on `master` after the release workflow runs.
- `.env`'s `VERSION_PLACEHOLDER` is unrelated to this contract. Both `android_release.yml` and `web_release.yml` overwrite it at build time, so the source of truth is **pubspec ↔ tag**, not `.env ↔ tag`.

`UpdateService` reads its current version via `PackageInfo.fromPlatform()`, which returns pubspec's `X.Y.Z`. If pubspec is stale (no release workflow ran after the last merge), the in-app updater sees the same version as the latest GitHub release and reports no update. The release workflow is what keeps pubspec fresh.

## In-app updater boundary

`_isNewerVersion` is a strict-greater semver compare. Pubspec currently ships at `1.0.0+1`. The first auto-release's baseline is pubspec's `1.0.0`; with a `chore:` (or other non-`feat`) merge, the next tag is `v1.0.1` (patch bump). Pubspec gets bumped to `1.0.1+2` in lockstep. The first release is **not** a no-op for the updater — `1.0.0` (installed) vs `1.0.1` (release) is strictly greater and the prompt fires.

If the merge is `feat:`-prefixed, the first release is `v1.1.0` (minor bump) and pubspec becomes `1.1.0+2`; the updater still detects a greater version and prompts. The boundary no-op only matters in a hypothetical future where a contributor manually edits pubspec down to match an existing tag — the resolver would then produce no bump because the baseline and the target align, and the first release would be a no-op. Don't manually edit pubspec to engineer that; it's an edge case the system already handles correctly via its strict-greater compare.

## Manual escape hatch

If you need to push a tag by hand (re-run after a fix, recover from a missed workflow run, or publish a backport):

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

The workflow treats HEAD as already-tagged when it sees `vX.Y.Z` pointing at it, so it becomes a no-op. To force a fresh tag anyway, delete the old one first:

```bash
git tag -d vX.Y.Z
git push origin :refs/tags/vX.Y.Z
```

## Token source

`pr_title_check` uses the default `GITHUB_TOKEN` with `pull-requests: write` permission granted at the job level. No additional secret is required. If the org restricts GITHUB_TOKEN write on PRs, the comment will 403 — fall back to manually renaming the PR before merging.
